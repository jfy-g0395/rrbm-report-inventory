| Column | Type | Notes |
|--------|------|-------|
| `id` | BIGSERIAL PK | |
| `expense_id` | BIGINT NOT NULL FK → expenses.id ON DELETE CASCADE | |
| `item_description` | VARCHAR(300) NOT NULL | **Free-text only — no category FK** |
| `amount` | NUMERIC(10,2) NOT NULL default 0 | |

### Entry flow

`POST /api/expenses` — `ExpenseController.java`
- Requires valid JWT (`Authorization: Bearer ...`); 401 if missing/invalid
- Body: `{ "date": "YYYY-MM-DD" (optional, defaults today), "items": [ { "itemDescription": "...", "amount": 0.00 }, ... ] }`
- No backdating restriction — any past date is accepted; there is **no backdating window setting**
- Writes one `Expense` + N `ExpenseItem` rows, logs `EXPENSE_RECORDED` to activity log
- Read: `GET /api/expenses?date=YYYY-MM-DD` and `GET /api/expenses/range?start=&end=`

**What does NOT exist yet (needed per E1–E3):**
- No category/sub-category table
- No void/audit columns on expenses (`voided_at`, `voided_by`, `void_reason`)
- No backdating-window setting in the `settings` table
- Expenses are **not** in the `transactions` ledger — they live only in `expenses`

## 4. User / role / permission system

### `users` table — `User.java`

Key fields for auth/access:

| Field | Notes |
|-------|-------|
| `role` | VARCHAR(20); DB constraint via `chk_role` |
| `allowed_pages` | TEXT — JSON array e.g. `["orders","inventory"]`; NULL = unrestricted (SUPER_ADMIN bypass) |
| `admin_security_key` | VARCHAR(255) BCrypt hash; used to authorise Cancel / Void / Force-close |
| `must_change_password` | BOOLEAN; cleared on own password change |
| `status` | `ACTIVE` / `INACTIVE` |

### Role names

Valid roles declared in `UserController.VALID_ROLES` (line 24–25):
```
SUPER_ADMIN | ADMIN | ADMINISTRATOR | ACCOUNTING | STAFF | STANDARD_USER
```

**⚠️ DB constraint mismatch:** `V11__user_roles.sql` only allows:
`SUPER_ADMIN`, `ADMIN`, `ADMINISTRATOR`, `STAFF`, `STANDARD_USER`.
`ACCOUNTING` is in the Java code's list but **NOT in the DB `chk_role` constraint**.
Inserting a user with `role = 'ACCOUNTING'` will throw a constraint violation.
Session E1 (or a standalone migration) must add `ACCOUNTING` to the constraint.

### Page-access system

`allowed_pages` is a JSON text array stored on each user. `NULL` = unrestricted (SUPER_ADMIN always bypasses). Frontend enforces page gating; backend does not.

Full page-key list (from `UserController.ALL_PAGES`, line 27–28):
```
orders, order-history, daily-reports, inventory, purchase-orders,
receive-stocks, rejected-items, reports, delivery-reports,
activity-log, employees, expenses, payables
```

**⚠️ Stale V14 seed:** `V14__expenses_and_page_permissions.sql` sets `allowed_pages` to only 8 keys for pre-V14 users. Six pages added since (`order-history`, `daily-reports`, `purchase-orders`, `rejected-items`, `payables` and effectively `suppliers`) are absent from those rows. Existing non-SUPER_ADMIN users may lack access to newer pages.

### Permission enforcement pattern

- All write endpoints extract user ID from `Authorization: Bearer <JWT>` via `JwtUtil.extractUserId()`
- Role-elevation, account disable, and page-permission changes: `SUPER_ADMIN` only (checked in `UserController`)
- Force-close daily report: dual-auth — caller's `adminSecurityKey` + any SUPER_ADMIN's `adminSecurityKey`
- No Spring Security role-based rules; `SecurityConfig` is `anyRequest().permitAll()` — auth is JWT-filter only

## 6. Accounting / cash-flow module

### Transaction ledger

**`transactions` table — `Transaction.java`**

| Column | Notes |
|--------|-------|
| `transaction_code` | VARCHAR(80) UNIQUE — e.g. `SALE-210526-000001`, `VOID-210526-000001-{ms}`, `COLL-DEFER-{orderId}` |
| `order_id` | VARCHAR(20) nullable FK |
| `transaction_type` | `SALE` (pos) / `REFUND` (neg) / `VOID` (neg) / `RETURN` (neg) / `DISCOUNT` (neg) / `ADJUSTMENT` (+/−) |
| `amount` | NUMERIC(12,2) — positive for SALE; negative for reversals |
| `reference_type` | `ORDER`, `EXPENSE`, `MANUAL` |
| `reference_id` | Free-form; orderId, expenseId, etc. |
| `effective_date` | DATE — the date this hits for reporting; post-close reversals use today's date to keep history immutable |
| `created_by` | BIGINT user ID |

**Formula:** `NET_SALES = grossSales + refundsTotal + adjustmentsTotal`  
where `refundsTotal = SUM(REFUND + VOID + RETURN)` (already negative).

### Service — `TransactionService.java`

| Method | Called by | Purpose |
|--------|-----------|---------|
| `recordSale(order, userId)` | `OrderService` | SALE on order creation; idempotent |
| `recordVoid(order, amount, userId, reason)` | `OrderService.cancelOrder()` | VOID on full cancellation; net basis |
| `recordReturnRefund(orderId, amount, reason, userId, userName)` | `OrderService.processReturn()` | RETURN on post-sale physical return |
| `recordItemVoid(orderId, amount, reason, userId, userName)` | `OrderService.voidOrderItems()` | VOID for partial item-level void |
| `recordAdjustment(orderId, amount, reason, userId, userName)` | `TransactionController POST /adjustment` | Manual correction |
| `recordDeferralVoid(order, userId)` | `DailyReportService` force-close | VOID on force-close; net basis; hits original order date |
| `recordCollectionSale(order, userId)` | `OrderService.collectOrder()` | SALE on deferred collection; hits original order date |

### Daily report close

`DailyReportService.closeDailySales()` — called by `DailyReportController POST /api/reports/close-daily`.  
Snapshot is immutable: `daily_reports` row stores `gross_sales`, `refunds_total`, `adjustments_total`, `net_sales`, `total_transactions`, plus operational stats (order counts by source, top product, etc.).

### What is NOT in the ledger

- **Expenses** — stored in `expenses` / `expense_items` only. No expense transaction type. Expense totals are queried directly from the `expenses` table in reports.
- **Payables** — supplier payables in the `payables` table (linked to `delivery_log`). Not in the transaction ledger.
- **Cash-flow widget** — does not exist yet. Plan A5 calls for Revenue − Expenses − Commissions widget.

### Settings table

`settings` table — `Settings.java` (key/value store). Currently seeded keys:
- `company_address` → `"116 Santan St., Fortune, Marikina City"`
- `company_contact` → `"+63 966 846 9993"`

No expense-backdating or commission-related settings exist yet.  
**Note:** `printOrderReceipt` in `app.js` hardcodes the address/contact (lines 787–788) rather than reading from the settings API.
