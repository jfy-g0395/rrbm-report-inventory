# Session 53 — Dark Mode Form Fix + Product-Analytics Dashboard

## Context
Two pieces: (1) a dark-mode bug where `<select>` dropdowns and their `<option>` items render unreadable (light text on default-white) on the New Order, Receive Stock, and Expenses pages; (2) a major dashboard expansion adding product-hierarchy analytics — pizza-box quota with period targets, total quantity sold, category & subcategory breakdowns, top products, and 5 charts — with weekly/monthly driven by **closed days only**.

Per user decisions: classification **reuses the existing inventory `category`/`sub_category` fields** (no new product fields, no migration), and the new sections are **added below** the existing financial cards (nothing removed).

---

## Part 1 — Dark Mode Form Fields (CSS only)

**Root cause:** `.form-select` (styles.css ~683) sets the dropdown-arrow `background-image` but **no `background-color` and no `color`**. A plain `<select class="form-select">` inherits the page's light `--text-primary` on the browser-default white control → invisible in dark mode. Native `<option>` popups also default to white.

**Fix (`css/styles.css`):**
- Give `.form-select` explicit `background-color: var(--input-bg);` and `color: var(--text-primary);` (keep arrow image; use `background-color`, not `background` shorthand).
- Dark-theme safety net:
```css
[data-theme="dark"] select,
[data-theme="dark"] .form-select,
[data-theme="dark"] .form-control,
[data-theme="dark"] input,
[data-theme="dark"] textarea { background-color: var(--input-bg); color: var(--text-primary); }
[data-theme="dark"] select option,
[data-theme="dark"] .form-select option,
[data-theme="dark"] .form-control option { background-color: var(--input-bg); color: var(--text-primary); }
```
- Sweep New Order (`view-new`), Receive Stock (`view-delivery`), Expenses (`view-expenses` + expense modal) for inline light bg/color on inputs/selects; rely on the classes/overrides above.

**Verify:** dark mode → all dropdowns/options/inputs readable on the three pages.

## Part 2 — Product-Analytics Dashboard

### Category mapping (reuse existing inventory categories — no migration)
Actual stored `products.category` / `sub_category`:
- `Pizza Box` (Claycoat/Generic/Plain), `Pizza Supplies` (blank sub)
- `Die-Cut Packaging Boxes` (Bilao/Flower/Mailer/Party/Roasted Food)
- `Offset Packaging Boxes` (Pastry Box)
- `Packaging Supplies` (Tapes/Wrappers), `RSC` (blank sub)

**Backend classifier** maps stored category → 4 spec buckets:
| Stored `category` | Dashboard category | Subcategory |
|---|---|---|
| `Pizza Box` | **Pizza Supplies** | `Pizza Boxes` |
| `Pizza Supplies` | **Pizza Supplies** | `Other Pizza Supplies` |
| `Die-Cut Packaging Boxes` | **Die Cut Packaging Boxes** | stored `sub_category` |
| `Offset Packaging Boxes` | **Offset Packaging Boxes** | `Pastry Boxes` |
| else (`Packaging Supplies`,`RSC`,null) | **Packaging Supplies** | stored `sub_category` or `Other` |

Pizza-box quota actual = qty of products in subcategory `Pizza Boxes` (stored category `Pizza Box`).

### New endpoint `GET /api/dashboard/product-analytics?period=daily|weekly|monthly`
- `daily` → today's live non-cancelled orders (`findByCreatedAtDateWithItems`).
- `weekly` → Mon→today, **closed days only**: closed dates from `dailyReportRepository.findByReportDateBetweenOrderByReportDateDesc(weekStart, today)`; aggregate `findByDateRangeWithItems` orders whose date ∈ closed set.
- `monthly` → 1st→today, same closed-days filter.
- Batch `productRepository.findAllById` for category lookup (reuse existing N+1 pattern in `getStats`).

Response: `{ period, periodStart, periodEnd, totalQtySold, pizzaQuota:{target,actual,pct}, categories:[{name,qty,pct,rank,subcategories:[{name,qty}]}], topProducts:[{name,qty}×5], trend:[{label,qty}] }`.
- Quota targets: daily 5000, weekly 30000, monthly = previous-month closed pizza-box total (`null` if none) → `pct=target?actual/target*100:null`.
- Trend: daily=24 hour buckets; weekly=Mon–Sun; monthly=each date — summing item quantities.

### Frontend (`index.html`, new sections below Row 5; refresh on `switchDashTab`)
1. Period-aware Pizza Quota KPI (target/actual/achievement% relabels per period).
2. Product Sales Summary (Total Quantity Sold).
3. Category Sales Breakdown table (Category | Qty | Contribution % | Rank).
4. Subcategory expandable rows per category.
5. Top 5 Products (period) — reuse top-5 styling.
6. Charts (Chart.js, reuse existing pattern): Sales Trend line (axis relabels per period), Category Distribution donut, Category Performance stacked bar.

### Frontend JS (`app.js`)
- `loadProductAnalytics(period)` — fetch, render summary/category/subcategory/top, build 3 charts (destroy-before-recreate via `appState`).
- Wire into `switchDashTab(period)` alongside existing `renderDashboard()`.
- Reuse `escapeHtml`, number formatting, existing Chart.js config style (`chart-sales`, `chart-source-donut`).

### Data update rule
Weekly/Monthly reflect closed days automatically: a date only appears once its `daily_reports` row exists (created at close). No scheduler needed.

## 50-A-1 · Dark Mode — Table backgrounds remain white

**Root cause:** `styles.css` defines `--bg-card` and `--table-stripe` for dark mode, but the `.table` element's `<td>` and `<tbody>` backgrounds may not be using CSS variables. Inline `style="background:#fafafa"` or `background:#fff` on striped rows ignores the theme.

**Fix (`css/styles.css`):**
- Add dark-mode override for table body backgrounds:
```css
[data-theme="dark"] .table tbody tr td,
[data-theme="dark"] .table tbody tr:nth-child(even) td {
  background: var(--bg-card);
  color: var(--text-primary);
}
[data-theme="dark"] .table thead th {
  background: var(--bg-secondary, #2d2d2d);
  color: var(--text-primary);
}
```
- Also check `.stat-card`, `.card-body`, and any inline `background:#fafafa` or `background:#f8f9fa` used in the monthly report cards — add dark overrides for those.

**Scope:** All tables across the app (Top 10 Products, Daily Order Summary, Daily Accounting Breakdown, etc.)

## 50-A-2 · Dashboard blank on first login

**Root cause:** In `doLogin()`, `navigateTo('dashboard')` is called which sets the view active, but `renderDashboard()` is called BEFORE `navigateTo()` — charts may render into a container that isn't visible yet (Chart.js needs visible containers for sizing).

**Fix (`app.js` — `doLogin()`):** Move `renderDashboard()`, `renderTopProductsToday()`, and `updateCollectionsBadge()` to AFTER `navigateTo('dashboard')`. The navigate function shows the view first, then the render calls populate it while it's visible.

Current order:
```
_clearSessionState();
navigateTo('dashboard');
renderDashboard();        // these run after navigateTo → view is visible
renderTopProductsToday();
updateCollectionsBadge();
```

This should already be correct from Session 45, but verify the actual execution order — `navigateTo` is synchronous, so it should work. If charts still fail, add a `setTimeout(renderDashboard, 100)` to let the DOM settle.

5. Alternatively: preload html2canvas in the main page `index.html` as a `<script>` tag so it's always available, then in the popup window use `opener.html2canvas` or just reference it directly.

**Simplest fix:** Add html2canvas CDN to `index.html` `<head>`, then in the popup window reference `window.opener.html2canvas || window.html2canvas`.

## Session 45 — Bug Fixes + Quick Wins

### Context
Several small bugs and UX gaps found across the app that can be resolved without migrations.

### 45-A · Session state not cleared on re-login
**Problem:** When a second user logs in, the first user's cached data (orders, inventory rows, etc.) remains visible in `appState` until a manual page refresh.  
**Fix (`app.js` — `doLogin()`):** After storing token/user, call a new `_clearSessionState()` helper that resets all appState array/object caches (`allOrders`, `cachedProducts`, `inventoryAllProducts`, `_importParsed`, `_collectionsParsed`, etc.) and clears any rendered table bodies that might show stale data, then explicitly calls `navigateTo('dashboard')`.

```javascript
function _clearSessionState() {
  appState.allOrders = [];
  appState.cachedProducts = [];
  appState.inventoryAllProducts = [];
  appState._importParsed = [];
  // clear rendered tables
  ['list-tbody','order-history-tbody','collections-tbody',
   'inventory-tbody','activity-log-tbody'].forEach(function(id){
    var el = $(id); if (el) el.innerHTML = '';
  });
}
```

### 45-B · Login always navigates to dashboard
**Fix (`app.js` — `doLogin()`):** After `_clearSessionState()`, explicitly call `navigateTo('dashboard')` so the dashboard nav item is active regardless of which view was last active before logout.

# Session 42 — Collections Page + Close Daily Override + Monthly Report PDF

## Context
Three linked features:
1. **Monthly Report PDF** is missing the non-pizza product breakdown and has no supplier payables section.
2. **Collection Page** (new) — orders where the customer didn't pay on delivery day need a dedicated page so admins can track and collect payment later. Any ACTIVE/PENDING order from any date where payment hasn't been received appears here.
3. **Close Daily Sales override** — currently the system blocks closing if any ACTIVE/PENDING orders exist. A dual-security-key override (logged-in admin + any Super Admin) is needed to force-close; those uncollected orders are moved to PENDING_COLLECTION status and excluded from today's revenue. When collected later, revenue is retroactively posted to the **original order date**.

**Key design decisions:**
- Revenue posts back to the original order date (Monday's report gets updated when Thursday's collection happens)
- Inventory is already deducted at order creation — no change needed
- SALE transactions are recorded only when an order is DELIVERED or collected; ACTIVE/PENDING orders at close time have no SALE transaction yet, which is why the current guard blocks the close
- Payables section in PDF = supplier delivery payables (PENDING, what RRBM owes to vendors) — not customer receivables

# Session 41 — CSV Import: Template Format + Smarter Matching

## Context
The current CSV import tries to parse raw ecommerce platform exports (Shopee, TikTok, Lazada) with two skipped header rows, inferred column positions, and name-only fuzzy matching. This is fragile. The business owner wants a controlled, downloadable template CSV that they fill in (mapping from any platform export), so parsing is deterministic and reliable. Additional needs: unit price sourced from CSV line total (not inventory price), a SKU column for future direct-match, better name-matching accuracy, and a "group by customer" option when the same buyer appears under multiple separate order numbers.

---

## Template Format

**Download template filename:** `rrbm-import-template.csv`

| Col | Index | Header | Usage |
|-----|-------|--------|-------|
| A | 0 | Order Number | Groups rows into one order; goes into Notes |
| B | 1 | Shipping | Courier name; used for platform detection + Notes |
| C | 2 | Tracking Number | Stored in Notes only; not a separate field in RRBM |
| D | 3 | Product Name | Primary matching field against inventory |
| E | 4 | Quantity | Line item qty |
| F | 5 | Total | Line total (₱); unitPrice = Total ÷ Qty |
| G | 6 | Customer Name | Buyer name for RRBM order |
| H | 7 | SKU | Optional now; future direct match to product `sku` / `productCode` / `itemCode` |

Row 1 = headers (skipped). Row 2+ = data. No date-header row (old format had 2 skipped rows).

---

| Behavior | Before | After |
|----------|--------|-------|
| Header rows skipped | 2 (date + headers) | 1 (headers only) |
| Customer name column | col[7] | col[6] |
| Unit price source | Inventory `unitPrice` | CSV Total ÷ Qty (editable in preview) |
| SKU column | Not present | col[7] — passed to matcher |
| Product matching | Name fuzzy only | SKU exact → name fuzzy fallback |

# Session 40 — Set / Bundle Products

## Context
Certain products (e.g. "Cochinillo Top" + "Cochinillo Bottom", "Bilao Box Top" + "Bilao Box Bottom") must be tracked individually in inventory but are **always sold together as a paired set**. A customer orders one "Cochinillo Set" at one price — but the system must deduct 1× Top and 1× Bottom from the same warehouse simultaneously. Components are never sold individually. This session adds full "set product" support: data model, stock deduction logic, inventory UI, and order form awareness.

---

## Confirmed Requirements
- Components (Top/Bottom) are **set-only** — never ordered individually
- When a set is ordered from WH1, **all components deduct from WH1**
- Bilao Box set = Top + Bottom (1:1 ratio)
- Cochinillo set = Top + Bottom (1:1 ratio)
- A set can have any number of components with custom `quantityPerSet` (supports future N-component sets)
- The set product has its own `unitPrice`; components have no individual cost/price requirement

## Effective Stock Calculation (frontend helper)

```javascript
function effectiveSetStock(setProduct) {
  var products = appState.cachedProducts || [];
  if (!setProduct.isSet || !setProduct.components || !setProduct.components.length) return null;
  var min = Infinity;
  setProduct.components.forEach(function(c) {
    var comp = products.find(function(p){ return p.id === c.componentProductId; });
    if (!comp) return;
    var avail = Math.floor((comp.stockWh1 + comp.stockWh2 + comp.stockWh3) / c.quantityPerSet);
    if (avail < min) min = avail;
  });
  return min === Infinity ? 0 : min;
}
```

# Session 39 — E-commerce CSV Order Import

## Context
Encoding e-commerce orders (Shopee, TikTok, Lazada) one-by-one is very tedious for the business owner. Each platform exports a CSV file daily containing all orders with order numbers, courier tracking, product names, quantities, and totals. This feature adds a **CSV import modal** so an entire day's e-commerce orders can be uploaded and imported in one action instead of being manually encoded.

---

## CSV Structure (from real Shopee/TikTok/Lazada export)

```
Row 1:  Date header (e.g. "May 30 2024") → skip
Row 2:  Column headers → skip
Row 3+: Order No | Shipping (courier) | Tracking Number | Product Name | Quantity (packs) | Units (total pieces) | Total (₱ per line) | Seller Name
```

**Key rules:**
- One row per product line. Orders with multiple products repeat the same Order No across rows.
- `Seller Name` (customer) is only filled on the **first row** of each order — blank on subsequent rows.
- `Quantity` (col 5) = number of packs/bundles ordered → this is the quantity to use in the order system.
- `Total` (col 7) = ₱ price for that specific line item.
- `Units` (col 6) = total pieces (e.g. 3 packs × 50pcs = 150) → not needed for order creation.

**Platform detection logic** (from Shipping column + Order No format):

| Shipping value | Platform |
|---|---|
| Contains "TikTok" | TIKTOK |
| Contains "LZD" | LAZADA |
| SPX | SHOPEE |
| YTO / Flash / J&T (with Shopee-format order No.) | SHOPEE |
| Fallback: Order No is 18+ digit number | TIKTOK |
| Fallback: Order No is 15 digit number | LAZADA |
| Fallback: Order No is alphanumeric | SHOPEE |

---

## Recommended Approach: Frontend CSV Parsing + Live Preview + Batch Backend Endpoint

**Why this approach:**
- No file storage on the server — CSV parsed entirely in the browser using `FileReader` + `split('\n')`
- User sees a live preview table before committing — can catch errors and fix unmatched products
- Product matching happens client-side against `appState.cachedProducts` (already loaded)
- One new backend endpoint `POST /api/orders/batch` handles bulk insert; reuses existing `OrderService` / `OrderController.createOrder()` logic

**Import defaults** (applied automatically, can be overridden in the modal header):
- Status: **ACTIVE** (orders still need to be packed/shipped)
- Payment mode: **COD** (most common for PH e-commerce; override available)
- Source: **ECOMMERCE** (always)
- Order type: **STANDARD**
- Warehouse: **auto** (highest available stock, same as manual order creation)
- Notes field: auto-populated with `"Tracking: {trackingNumber} via {courier}"`

## Product Name Matching Algorithm (`matchCsvProduct(csvName)`)

1. Normalize both: `lower().replace(/[^a-z0-9\s]/g, ' ').trim()`
2. Extract meaningful tokens: remove common filler words ("pcs", "bundle", "pack", "box", "corrugated", "inches", etc.)
3. Score each product in `appState.cachedProducts` by token overlap
4. Return best match if score ≥ threshold (2+ meaningful tokens match)
5. Return `null` (unmatched) if score < threshold

**Examples from real CSV data:**
- `"Pizza Box Corrugated - Care for a slice Gen4/ 12 inches/ 50pcs"` → matches product containing "gen4" + "12"
- `"Grease Proof Wax Paper | 10x10 100pcs"` → matches product containing "10x10" or "wax paper 10"
- `"Clay Coated Pizza Box | Cg2 Black | 10 inches (50 pcs)"` → matches "cg2" + "black" + "10"

## Backend — New Batch Endpoint

**File:** `OrderController.java`

```java
@PostMapping("/batch")
public ResponseEntity<?> createOrderBatch(@RequestBody List<CreateOrderRequest> requests,
                                           @RequestHeader("Authorization") String authHeader) {
    // Extract token, validate user
    int imported = 0, failed = 0;
    List<Map<String, String>> errors = new ArrayList<>();
    for (CreateOrderRequest req : requests) {
        try {
            createOrder(req, authHeader);  // delegate to existing single-order logic
            imported++;
        } catch (Exception e) {
            failed++;
            errors.add(Map.of("orderRef", req.getNotes() != null ? req.getNotes() : "?", "reason", e.getMessage()));
        }
    }
    return ResponseEntity.ok(Map.of("imported", imported, "failed", failed, "errors", errors));
}
```

- No new migration needed — orders are stored in the same `orders` table
- Reuses all existing validation, stock deduction, activity logging from `createOrder()`
- Partial success allowed — a bad row doesn't block the rest

## Detailed `app.js` Function Descriptions

### `parseCsvFile(file)`
- Reads file via `FileReader.readAsText()`
- Splits by newline, skips row 1 (date header) and row 2 (column headers)
- Parses remaining rows: split by comma, but handle quoted fields
- Groups rows by Order No (`groupCsvRows()`)
- Returns array of order objects

### `groupCsvRows(rows)`
- Groups by Order No (col 0)
- First row in group = `customerName` (Seller Name, col 7)
- All rows in group = `items[]`
- Order total = sum of all line totals in the group
- Calls `detectPlatform()` to set `ecommercePlatform`

### `detectPlatform(orderNo, shipping)`
- Returns "SHOPEE", "TIKTOK", or "LAZADA" using the logic table above

### `matchCsvProduct(csvName)`
- Scores all products in `appState.cachedProducts`
- Returns `{ product, confidence }` — confidence: "high" | "low" | null

### `renderImportPreview(parsedOrders)`
- Builds the preview table rows
- Each order row: collapsed by default, expandable
- Each item row: product name | match dropdown | qty | unit price | line total
- Calls `updateImportSummary()` to update footer counts

### `submitCsvImport()`
- Collects all "ready" orders from `_importParsed`
- Builds `CreateOrderRequest[]` array
- POST to `/api/orders/batch`
- On success: close modal, reload order list, show toast "X orders imported"
- On partial failure: show error list in a summary section

# Bug Fix — Delivery Receipt Number VARCHAR(10) Too Short

## Context
When receiving stock, users cannot enter an 11-digit PO number as the DR number. The `delivery_log.receipt_number` column was defined as `VARCHAR(10)` in migration V5. Frontend and backend validation already accept 2–20 alphanumeric characters, and the HTML input has `maxlength="20"` — so the restriction is purely at the DB layer. Any DR number longer than 10 characters is rejected by PostgreSQL on INSERT.

## Root Cause — Layer Audit
| Layer | Current Setting | Status |
|---|---|---|
| DB `delivery_log.receipt_number` (V5 migration) | `VARCHAR(10)` | ❌ Too short |
| Backend regex `ProductController.java` | `^[A-Za-z0-9\-]{2,20}$` | ✅ OK |
| Frontend regex `submitDeliveryReceipt()` | `/^[A-Za-z0-9\-]{2,20}$/` | ✅ OK |
| HTML input `maxlength` | `maxlength="20"` | ✅ OK |
| `payables.receipt_number` (V17) | `VARCHAR(20)` | ✅ OK |
| `po_items.dr_number` (V23) | `VARCHAR(100)` | ✅ OK |

## Fix — One New Migration File

**Create:** `D:\ClaudeProjects\rrbm_daily\rrbm-backend\src\main\resources\db\migration\V24__widen_delivery_receipt_number.sql`

```sql
-- Widen delivery_log.receipt_number from VARCHAR(10) to VARCHAR(30).
-- Fixes: 11-digit PO numbers (and DR numbers > 10 chars) rejected at DB level.
-- Frontend/backend validation already allows 2-20 chars; DB must be at least as wide.
ALTER TABLE delivery_log ALTER COLUMN receipt_number TYPE VARCHAR(30);
```

VARCHAR(30) chosen (not 20) to give headroom beyond the current 20-char validation limit.

**No other files need changing.**
